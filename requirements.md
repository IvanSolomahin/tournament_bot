# Telegram-бот для регистрации на турнир по парламентским дебатам

## 1. Общее описание

Telegram-бот предназначен для регистрации команд на турнир по парламентским дебатам, сопровождения заявок, сбора подтверждений участия, контроля оплаты и организационной коммуникации с участниками.

Система поддерживает две роли:
- участник команды;
- администратор.

Команда состоит из двух участников, которые имеют одинаковые права в рамках своей заявки.

Оба участника могут:
- активировать бота;
- подтверждать привязку к команде;
- изменять название команды в пределах разрешённого этапа;
- отправлять подтверждение оплаты;
- проходить check-in по дням турнира.

Администратор может:
- просматривать заявки;
- вручную распределять команды между основным списком и вейтлистом;
- выполнять рассылки;
- формировать экспортные выгрузки.

## 2. Бизнес-правила

- Команда включает двух участников.
- Для каждого участника отдельно указываются имя, контакт, клуб и достижения при наличии.
- Участники одной команды могут принадлежать к разным клубам.
- Первый участник при создании заявки указывает username второго участника в Telegram.
- Заявка считается завершённой только после того, как оба участника самостоятельно активируют бота и подтвердят участие в команде.
- Оба участника имеют равные права на изменение названия команды до перевода заявки в основной список или в вейтлист.
- После перевода заявки в основной список или в вейтлист изменение названия команды не допускается.
- Распределение команды в основной список или в вейтлист осуществляется администратором вручную.
- Функция оплаты доступна только командам, включённым в основной список.
- Для команд из вейтлиста возможность оплаты должна отсутствовать.
- Для подтверждения оплаты каждый из двух участников отдельно отправляет своё подтверждение оплаты в бот.
- После получения двух подтверждений оплаты оплата считается подтверждённой автоматически.
- Ручная проверка подтверждений оплаты администратором не требуется.
- Бот должен отправлять организационные сообщения со ссылками на канал и чат турнира.
- Организатор может вручную запускать рассылку пользователям, которые не выполнили условия текущего этапа, например не подтвердили участие, не отправили подтверждение оплаты или не прошли check-in.

## 3. Основной сценарий работы

1. Один из участников активирует бота и создаёт заявку команды.
2. При создании заявки участник указывает название команды, данные двух участников и username второго участника в Telegram.
3. Система сохраняет заявку в статусе `ожидает подтверждения участников`.
4. Второй участник самостоятельно активирует бота.
5. После активации бота система предлагает второму участнику подтвердить участие в команде, в которой он указан по username.
6. После подтверждения обоих участников система переводит заявку в статус `зарегистрирована`.
7. Администратор рассматривает зарегистрированные заявки и вручную переводит команду в основной список или в вейтлист.
8. Команды из основного списка получают доступ к подтверждению участия и этапу оплаты.
9. Команды из вейтлиста не получают доступ к этапу оплаты.
10. На этапе оплаты каждый из двух участников отдельно отправляет подтверждение оплаты.
11. После получения двух подтверждений оплаты система автоматически фиксирует оплату как подтверждённую.
12. Перед турниром администратор может запускать организационные рассылки.
13. В первый и второй день турнира бот запрашивает подтверждение присутствия.

## 5. Функциональные требования

### 5.1. Управление пользователями

- **FR-5.1.1** Система должна активировать пользователя после получения команды `/start` в Telegram-боте.
- **FR-5.1.2** Система должна сохранять Telegram ID активированного пользователя.
- **FR-5.1.3** Система должна связывать Telegram ID пользователя с записью участника команды после подтверждения участия в команде.
- **FR-5.1.4** Система должна различать роли `участник команды` и `администратор`.
- **FR-5.1.5** Система должна определять административные права пользователя по Telegram ID, указанному в конфигурационном файле приложения.

### 5.2. Создание и регистрация команды

- **FR-5.2.1** Система должна позволять активированному участнику создать заявку команды.
- **FR-5.2.2** Система должна позволять указать название команды при создании заявки.
- **FR-5.2.3** Система должна позволять указать данные двух участников команды при создании заявки.
- **FR-5.2.4** Система должна требовать для каждого участника ввод имени и контакта.
- **FR-5.2.5** Система должна позволять для каждого участника указать клуб.
- **FR-5.2.6** Система должна позволять для каждого участника указать достижения.
- **FR-5.2.7** Система должна позволять участникам одной команды принадлежать к разным клубам.
- **FR-5.2.8** Система должна требовать указания username второго участника Telegram при создании заявки.
- **FR-5.2.9** Система должна сохранять новую заявку в статусе `ожидает подтверждения участников`.

### 5.3. Подтверждение состава команды

- **FR-5.3.1** Система должна требовать самостоятельную активацию бота каждым участником команды.
- **FR-5.3.2** После активации бота пользователем, username которого указан в заявке, система должна предложить ему подтвердить участие в соответствующей команде.
- **FR-5.3.3** Система должна подтверждать привязку участника к команде только после явного действия пользователя в интерфейсе бота.
- **FR-5.3.4** Система должна переводить заявку в статус `зарегистрирована` только после подтверждения участия обоими участниками.

### 5.4. Редактирование заявки

- **FR-5.4.1** Система должна предоставлять обоим участникам команды равные права на редактирование заявки.
- **FR-5.4.2** Система должна позволять обоим участникам изменять только название команды.
- **FR-5.4.3** Система должна позволять изменять название команды только до перевода заявки в статус `основной список` или `вейтлист`.
- **FR-5.4.4** После перевода заявки в статус `основной список` или `вейтлист` система должна запрещать изменение названия команды.

### 5.5. Управление статусом заявки

- **FR-5.5.1** Система должна поддерживать статусы заявки: `ожидает подтверждения участников`, `зарегистрирована`, `основной список`, `вейтлист`, `участие подтверждено`, `ожидает оплаты`, `частично оплачена`, `оплата подтверждена`, `снята`.
- **FR-5.5.2** Система должна автоматически изменять статус заявки при наступлении событий, определённых бизнес-правилами.
- **FR-5.5.3** Система должна предоставлять администратору возможность вручную изменять статус заявки.
- **FR-5.5.4** Система должна отображать участникам текущий статус их заявки.

### 5.6. Управление основным списком и вейтлистом

- **FR-5.6.1** Система должна предоставлять администратору возможность вручную переводить команду в статус `основной список`.
- **FR-5.6.2** Система должна предоставлять администратору возможность вручную переводить команду в статус `вейтлист`.
- **FR-5.6.3** Система не должна предоставлять командам из вейтлиста доступ к этапу оплаты.

### 5.7. Подтверждение участия

- **FR-5.7.1** Система должна предоставлять команде из основного списка возможность подтвердить участие в турнире.
- **FR-5.7.2** Система должна сохранять факт подтверждения участия команды.
- **FR-5.7.3** После подтверждения участия команда должна переводиться в статус `участие подтверждено`.

### 5.8. Подтверждение оплаты

- **FR-5.8.1** Система должна предоставлять этап оплаты только командам из основного списка.
- **FR-5.8.2** Система должна позволять каждому из двух участников команды загрузить отдельное изображение, используемое как подтверждение оплаты.
- **FR-5.8.3** Система должна сохранять факт получения подтверждения оплаты отдельно по каждому участнику.
- **FR-5.8.4** После получения подтверждения оплаты от одного участника система должна переводить заявку в статус `частично оплачена`.
- **FR-5.8.5** После получения подтверждений оплаты от обоих участников система должна автоматически переводить заявку в статус `оплата подтверждена`.
- **FR-5.8.6** Система не должна требовать ручной проверки подтверждений оплаты администратором.

### 5.9. Организационные сообщения

- **FR-5.9.1** Система должна отправлять участникам организационные сообщения со ссылками на канал и чат турнира.
- **FR-5.9.2** Система должна предоставлять администратору возможность вручную запускать отправку организационного сообщения.
- **FR-5.9.3** При запуске рассылки система должна отправлять уведомление пользователям, которые не выполнили условия текущего этапа, включая случаи, когда пользователь не подтвердил участие, не отправил подтверждение оплаты или не прошёл check-in.

### 5.10. Подтверждение присутствия

- **FR-5.10.1** Система должна обеспечивать подтверждение присутствия участников в первый день турнира.
- **FR-5.10.2** Система должна обеспечивать подтверждение присутствия участников во второй день турнира.
- **FR-5.10.3** Система должна сохранять результат подтверждения присутствия отдельно для каждого участника и отдельно для каждого дня турнира.

### 5.11. Уведомления и напоминания

- **FR-5.11.1** Система должна автоматически отправлять напоминания о необходимости завершить регистрацию команды.
- **FR-5.11.2** Система должна автоматически отправлять напоминания о необходимости подтвердить участие в турнире.
- **FR-5.11.3** Система должна автоматически отправлять напоминания о необходимости отправить подтверждение оплаты.
- **FR-5.11.4** Система должна автоматически отправлять напоминания о необходимости пройти check-in.
- **FR-5.11.5** Система должна отправлять напоминания только пользователям, у которых соответствующее действие ещё не выполнено.

### 5.12. Административные рассылки

- **FR-5.12.1** Система должна предоставлять администратору возможность отправлять сообщение одной команде.
- **FR-5.12.2** Система должна предоставлять администратору возможность отправлять сообщение группе команд.
- **FR-5.12.3** Система должна поддерживать выбор получателей по статусу заявки.

### 5.13. Просмотр и обработка заявок администратором

- **FR-5.13.1** Система должна предоставлять администратору список заявок команд.
- **FR-5.13.2** Система должна предоставлять администратору карточку команды.
- **FR-5.13.3** Карточка команды должна содержать данные участников.
- **FR-5.13.4** Карточка команды должна содержать статус заявки.
- **FR-5.13.5** Карточка команды должна содержать статус оплаты.
- **FR-5.13.6** Карточка команды должна содержать историю ключевых действий.

### 5.14. Экспорт данных

- **FR-5.14.1** Система должна предоставлять администратору возможность экспортировать данные по заявкам в формате XLSX.
- **FR-5.14.2** Система должна включать в экспорт название команды.
- **FR-5.14.3** Система должна включать в экспорт данные участников.
- **FR-5.14.4** Система должна включать в экспорт данные о клубах участников.
- **FR-5.14.5** Система должна включать в экспорт данные о достижениях участников.
- **FR-5.14.6** Система должна включать в экспорт статус заявки.
- **FR-5.14.7** Система должна включать в экспорт статус оплаты.
- **FR-5.14.8** Система должна включать в экспорт данные о подтверждении присутствия по каждому дню турнира.
