# Tournament Bot

Telegram-бот для регистрации команд на турнир по парламентским дебатам: заявки команд из двух участников, подтверждение участия, оплата, check-in в дни турнира и инструменты администратора. Полный список юзкейсов и бизнес-правил — в [requirements.md](requirements.md).

## Стек

- Python 3.8+, [aiogram](https://docs.aiogram.dev/) 3.x (long polling)
- SQLite (через `sqlite3` + asyncio executor, WAL-режим)
- APScheduler — напоминания и check-in промпты в том же процессе
- openpyxl — экспорт заявок в XLSX
- Фото подтверждения оплаты хранятся только как Telegram `file_id`, без записи на диск

Архитектура рассчитана на слабый сервер (память/диск в обрез): один процесс, без вебхуков, без отдельной БД, без локального хранения медиа.

## Установка

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Заполните `.env`:

| Переменная | Описание |
|---|---|
| `BOT_TOKEN` | токен бота от @BotFather |
| `ADMIN_IDS` | Telegram ID администраторов через запятую |
| `CHANNEL_LINK` / `CHAT_LINK` | ссылки на канал/чат турнира (для ручной вставки в сообщения админом) |
| `DB_PATH` | путь к файлу SQLite |
| `TOURNAMENT_DAY1` / `TOURNAMENT_DAY2` | даты турнира в формате `YYYY-MM-DD`, используются для check-in |

## Запуск

```bash
python run.py
```

При старте бот сам устанавливает меню команд в Telegram (`/` рядом с полем ввода) — отдельное для участников и расширенное для администраторов.

## Команды

**Участник:**
- `/create_team` — создать заявку команды (пошаговый диалог)
- `/status` — статус своей заявки
- `/rename_team` — изменить название команды (до распределения в основной список/вейтлист)
- `/checkin` — подтвердить присутствие в день турнира
- `/cancel` — отменить текущий диалог
- `/help` — справка

Подтверждение оплаты отправляется фотографией прямо в чат с ботом.

**Администратор** (только для ID из `ADMIN_IDS`):
- `/admin_applications` — список заявок с пагинацией, карточка команды (данные участников, статус оплаты, история действий, чеки оплаты), ручное распределение в основной список/вейтлист
- `/admin_broadcast` — рассылка одной команде или группе по статусу заявки
- `/admin_org_message` — произвольное организационное сообщение всем активным участникам
- `/admin_remind` — напомнить только тем, кто не выполнил конкретный этап (регистрация/участие/оплата/check-in)
- `/admin_export` — выгрузка всех заявок в XLSX

## Основной флоу

1. Один участник создаёт заявку (`/create_team`), указывает данные обоих и username второго.
2. Второй участник открывает бота (`/start`) и подтверждает участие — заявка переходит в статус «зарегистрирована».
3. Администратор распределяет заявку в основной список или вейтлист.
4. Команды основного списка подтверждают участие в турнире.
5. Каждый участник присылает фото подтверждения оплаты; статус автоматически меняется на «оплата подтверждена», когда оплатили оба.
6. В дни турнира участники подтверждают присутствие через `/checkin`.

На каждом шаге оба участника команды получают уведомления — независимо от того, кто из них совершил действие.

## Структура проекта

```
bot/
  main.py            — entrypoint, сборка бота/диспетчера/планировщика
  config.py          — загрузка .env
  models.py          — TeamStatus enum, дата-модели
  states.py          — FSM-состояния (регистрация, переименование, рассылка)
  keyboards.py       — инлайн-клавиатуры
  db/
    schema.sql       — таблицы users, teams, members, action_log
    database.py      — async-обёртка над sqlite3
    repo.py          — слой доступа к данным
  services/
    status_engine.py — единая точка переходов статуса команды
    notifications.py — отправка сообщений с ретраями, рассылки
    reminders.py      — джобы APScheduler
    export_xlsx.py    — сборка XLSX-выгрузки в памяти
  handlers/
    common.py, start.py, registration.py, confirmation.py,
    status.py, team_edit.py, participation.py, payment.py, checkin.py
    admin/
      applications.py — список заявок, карточка, распределение по спискам
      broadcast.py     — рассылки и напоминания
      export.py        — /admin_export
```

## Память и диск

Однопроцессная архитектура с `MemoryStorage` для FSM (стейты живут в RAM, не персистентны между перезапусками) и SQLite в WAL-режиме. На тесте процесс занимает ~150MB RSS. Медиа не пишутся на диск — только `file_id` в БД.
